# Support

Real A11y is maintained by one person on a best-effort basis. This page sets expectations so you know what to expect when you open an issue or ask for help.

## What goes where

| You want to…                              | Go here                                                                                                       |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Ask a question about how to use Real A11y | [Discussions → Q&A](https://github.com/real-a11y/real-a11y-dev/discussions/categories/q-a)                    |
| Report a bug with a reproduction          | [Issues — Bug report](https://github.com/real-a11y/real-a11y-dev/issues/new?template=bug_report.md)           |
| Propose a new feature or API change       | [Issues — Feature request](https://github.com/real-a11y/real-a11y-dev/issues/new?template=feature_request.md) |
| Report a security vulnerability           | See [SECURITY.md](./SECURITY.md) — **do not** file a public issue                                             |
| Report Code of Conduct concerns           | See [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md)                                                                |
| Show what you built                       | [Discussions → Show & Tell](https://github.com/real-a11y/real-a11y-dev/discussions/categories/show-and-tell)  |

## Response expectations

- **Security reports** — acknowledged within 72 hours, with a plan communicated back to you inside 7 days.
- **Accessibility regressions** (`label:a11y` on issues) — triaged within 72 hours, prioritised over new features.
- **Bugs with a clear reproduction** — typically triaged within a week.
- **Feature requests** — read within two weeks, but may sit open for longer while we decide if they fit the direction of the project.
- **Questions in Discussions** — answered when time permits. If nobody has answered in a week and you're blocked, feel free to ping.

I am one person. Silence is not indifference — it's usually me catching up. Nudging after a week is fine; nudging after a day is not.

## How to help us help you

Issues close faster when they include:

1. **A minimal reproduction.** StackBlitz / CodeSandbox / tiny repo / gist — anything we can run. Screenshots are fine for visual bugs but not a substitute for code.
2. **Version information.** `@real-a11y-dev/<package>` version, Node version, browser (if relevant), OS.
3. **What you expected vs. what happened.** Even one sentence each.
4. **What you already tried.** Saves us from suggesting things you've ruled out.

The issue templates walk you through this — please fill them in rather than opening blank issues.

## Contributing back

Bugs you can fix yourself are the best kind. [CONTRIBUTING.md](./CONTRIBUTING.md) covers the development setup, the test expectations, and the PR conventions. Small, focused PRs get reviewed faster than big ones.
