  heading "Why this split" (level 2)
  heading "Engine separate from renderer" (level 3)
  paragraph "Everything useful about the extraction engine (tree walk, role map, accessible-name computation, tab-sequence derivation, action dispatch) is framework-agnostic tree manipulation. Keeping it in with no UI dependency lets the testing package, the Playwright adapter, and downstream tooling use the engine without pulling Preact into their bundle."
    code "@real-a11y-dev/core"
  heading "React-specific concerns live in @real-a11y-dev/react" (level 3)
    code "@real-a11y-dev/react"
  paragraph "React 18 concurrent-mode safety requires ; SSR (Next.js App Router) requires a mount-gated portal; React 19's jsx-runtime has internals that don't live in React 18. All of that React-specific complexity is isolated to one package with a clear React peer dep. Vanilla and Vue projects pay none of that cost — they use directly."
    code "useSyncExternalStore"
    code "inspector"
  heading "Testing is fully headless" (level 3)
  paragraph "is used by CI jobs that run in Node with no browser, by Vitest unit suites with jsdom, and by Playwright E2E jobs with a real Chromium. One API, three runtimes. Decoupling the UI makes that possible — the tree view isn't involved in any assertion path."
    code "@real-a11y-dev/testing"
  heading "The findings engine has one home" (level 3)
  paragraph "A finding — "this button has no accessible name" — is defined and detected in exactly one place: . The type, the rule set, , and the primitives all live there, depending on nothing but . Everything downstream only what the engine produces: re-exports it for test authors, and / format it for agents and the shell. Keeping detection separate from presentation means a new rule is written once and every surface reports it, and a production package like never has to pull in a test-helper package to reach the engine."
    code "@real-a11y-dev/audit"
